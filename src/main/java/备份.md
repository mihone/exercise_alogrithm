# 做法
## 新版
解决方案就是标准化
标准化就是建立一套标准的流程来实现复杂检索条件的迭代开发。
具体可以拆分两点 第一点是配置化
配置化是把字段间的复杂逻辑规则化，通过规则计算，得出商品是否满足规则，最终形式是生成或者消除一个标签。
这样就把业务上复杂的检索诉求转化成了对标签的简单化检索。
在配置规则上，有两种形式，一种是相对简单的逻辑，通过aviator表达式来配置，一般由研发自己创建。
另一种是通过标签的管理系统，在页面上创建标签并维护标签的规则，这个可以由产品或者运营来创建。
通过配置化的形式解决烟囱式开发问题，减少了开发的成本。

第二点自动化，分为两种情况，一种是监听变更消息，当上游，外部系统来的数据源发生变更，比如商品变更，质量分变更等等，会自动根据规则重新计算标签，判断打标消标，进行更新
这里也包括上游业务变更重新刷数等场景，我们是通过消息监听的变更，因此上游的变化我们会自动适配计算更新。
另一种情况，一般在刚创建新标签的场景，创建后会自动的监听到变更，从doris里重新圈选数据，进行打标消标。
这两个场景总和起来满足标签变更的自动打标消标，实现自动化流程。

## 之前
在做法上，业界支持复杂条件检索的中间件有ES，和分析型数据库，比如clickhouse，doris，从即时性，使用场景考虑，选用了ES作为检索中间件。
然后就产生了一个问题，通过什么形式和流程把数据写入es，一种方案其实是监听商品变更直接写入，这样设计虽然简单，但由于商品变更消息量大，且峰值高，高峰值对es
性能会有一定影响，另外后面如果出现问题，很难排查。需要有个其他存储介质来协助，这里也是参考了cqrs架构，先根据商品变更异构一份商品信息，再监听异构的数据的binlong
写入es，好处有三点，1，异构过程中可以合并一些同一商品变更消息，降低流量，2，方便后面做数据校验，问题排查。
在如何设计检索条件的存储结构上，综合考虑维护便利和有es查询语句复杂嵌套带来的查询性能因素，摈弃了原来索引的横向设计，
将检索条件转化为标签，通过标签的交并补逻辑，以及标签的层级关系，完成对商品的检索。
这样问题转变成了如何设计标签的生产流程，可以支持快速迭代，开发维护成本低。
解决这个的思路有两个方向，配置化和自动化。
第一个 ，考虑标签本身是个比较通用的概念 很自然想到配置化，，而从配置的人角度考虑，可以使开发，可以使产品运营，从这两类人群考虑，设计了两个配置模式，一种是页面提供多种选项，满足条件即可打标，实现形式是根据条件进行数据圈选，
主要面向产品运营。
另一种面向开发，通过aviator表达式，支持复杂的逻辑条件，满足需要上下文信息来进行复杂判断的场景。
第二个，自动化，这涉及了整个系统数据的流转形式和流程。整个商品检索系统有三个模块，圈选模块，标签生产模块，数据读写模块。
标签生产模块会监听其他系统的kafka消息，基于上面说的配置化的标签规则，生产标签。生产的标签存入db，并将产生binlog以消息的形式对外通知。
数据读写模块则监听商品信息和标签信息的变更，写入es。

这里面有两个点，配置变更和外部数据变更如何实现的自动化。
首先，我会监听配置变更的binlog，配置变更会触发数据重新圈选，然后圈选的结果消息形式通知标签生产模块，标签生产模块组装上下文重新计算标签。
然后输出最终的商品标签结果，商品标签结果再通知数据读写模块进行es的更新。
第二个点，外部数据变更，比如说hive 的更新，其他系统产出的数据更新，我都统一使用消息来接收，这样外部的更新我们无须关注，只要外部系统发出变更通知
我会自动重算标签，更新es。

首先应用于连锁商家的商品管理，并完成与质量系统对接打通， 在质量域上为商家的商品质量提高赋能。
后面预期商家的商品分析等方向。
  

为什么选用cqrs架构
选择CQRS主要是为了解决写入瓶颈和数据模型冲突这两个核心问题。商品数和标签，如果直接写ES，
高并发写入会瞬间打满ES的索引资源，严重影响查询性能。CQRS把命令（数据变更）和查询（构建索引）分开，
让写入走MySQL，然后通过监听binlog异步同步到ES，这样就把实时写入的压力从ES转移到了db，
而db处理这种事务性写入更擅长。具体到批量写入和优先级策略，我们是在消费binlog消息后，
不是一条一条写ES，而是先按商家维度在内存里攒一批，比如攒够1000条或者每隔1秒，再批量提交一次。
同时，我们根据业务重要性区分了优先级，比如商家修改的，优先级最高，会走实时队列，尽快同步；像价格库存相关标签，十分敏感，需要尽快同步
而一些描述性信息的更新优先级就低一些，走延时队列。这样既保证了核心数据的及时性，又平滑了写入流量，最终把TP99控制在了1秒内。

为什么要按照商家维度拆索引
在美团建设商品检索系统时，我们选择按商家维度拆分索引，主要基于业务特点。我们的核心业务场景是连锁商家管理自己的商品，
查询也大多集中在单个商家维度内。这样拆分后，每次查询只需命中一个或少数几个索引，能极大减少扫描的数据量，提升查询效率。
路由机制上，我们基于商家ID进行哈希取模，将请求路由到对应的索引分片。对于跨商家查询，我们设计了聚合查询层，
可以并发查询多个索引再合并结果。收益很明显，TP99查询耗时从之前的几秒降到了1秒内。
潜在问题主要是索引数量多了管理起来麻烦，我们通过自动化脚本和监控告警来管理索引的创建、扩容和清理。另外就是得预防数据倾斜，
我们定期分析各商家数据量，对超大商家会考虑进一步拆分其索引。

如何保证可靠性
整个同步链路是：MySQL主库产生binlog，Canal server伪装成从库进行订阅和解析，解析后的数据变更事件会发送到mq，
然后由我们自研的消费服务消费消息，再批量写入ES。这样设计主要是为了解耦和缓冲，避免Canal或ES抖动直接影响对方。
为了保证可靠性，我们做了几件事：一 监听binlog中间件本身会持久化消费位点，即使重启也能从断点继续；
二是消息队列保证了消息不丢；三是消费服务有完整的重试和死信队列机制。同步延迟我们确实遇到过，一次是因为某个大商家批量更新商品，
产生了大量binlog，导致解析和消息堆积。我们通过优化消费服务的批量写入策略，增加处理并发度，并紧急扩容了消息队列的分片数来解决。
平时我们也有监控大盘，会实时跟踪binlog位点延迟和消息堆积情况。

标签配置这块怎么做的
在我们美团商品检索系统的项目中，元数据驱动和Aviator脚本都是为了提升检索条件配置的灵活性。
元数据驱动的方式，主要是把检索条件抽象成字段名、操作符、值类型等配置信息，存储在一张配置表里。
这种方式适用于逻辑相对固定的条件，比如商品状态、上架时间范围等，配置简单直观，业务方通过后台页面就能操作。
而Aviator脚本则更灵活，它允许业务方或研发同学编写表达式脚本，比如一些复杂的、需要多字段组合计算或者动态逻辑判断的条件，
比如根据促销活动和库存情况动态计算一个排序权重。脚本的优点是对研发来说熟悉，简洁，但缺点是需要一定的开发能力，并且要关注脚本的执行性能和安全，比如避免死循环。
我们在实践中会有一个评估，对于常规的、标准化的筛选就用元数据，对于需要快速试错或逻辑特别复杂的场景，就用脚本，
同时会对脚本进行基础的语法检查和执行超时控制。

行中反馈怎么用的缓存。
在携程的行中反馈系统里，我们根据数据的不同特性分场景使用了缓存。
Redis主要用来缓存用户提交的反馈数据本身。因为这部分数据来自不同用户，需要在服务集群间共享，并且写入后需要持久化到MySQL，
同时接口响应要求高，所以用Redis做一层缓存非常合适，我们设置了合理的过期时间，并通过定时任务清理过期数据。
而本地缓存，我们用的是类似Caffeine的组件，主要缓存的是M端配置系统里定义的反馈问题项，
比如问题分类、多语言文案等。这部分数据的特点是：读取频率极高（每次用户打开反馈页面都要拉取），
但变更频率很低（可能几天或几周才由运营改一次），并且对一致性要求不是那么实时（允许短时间延迟）。
用本地缓存可以避免每次请求都去查库或访问Redis，极大降低响应延迟和网络开销。
我们通过一个异步线程定时（比如每分钟）去拉取最新的配置数据来更新本地缓存，即使拉取失败，也有旧数据兜底，不影响核心功能。
这样设计，核心依据就是数据的热度、变更频率以及对一致性的要求。

结合项目，cqrs适合解决哪一类问题
从我负责的两个项目来看，CQRS最适合解决的是读写负载差异巨大、且查询模式非常复杂的系统问题。
比如商品检索系统，写入是批量同步商品数据，但查询需要支持几十亿数据、多标签组合的秒级检索，传统CRUD用一个模型很难兼顾。
引入CQRS后，我们最大的挑战是开发思维和运维习惯的转变。开发上，团队需要理解命令和查询的边界，设计好数据同步链路，
比如我们用binlog监听同步到ES。运维上，监控变得复杂，需要同时关注MySQL的写入延迟和ES的数据新鲜度、查询性能。
另外，数据一致性的保证也是个关键点，我们设计了实时和离线双校验流程来兜底。
虽然初期成本高，但长远看，它让系统在应对复杂查询和未来业务变化时更有弹性。

项目介绍
随着连锁商家占比提升，连锁商家的商品管理越来越重要，业务为了让商家更容易地寻找和管理自己的商品，
增加多维度复杂检索条件来支持，然而原有检索能力无法支持复杂条件检索，且硬编码导致迭代成本极高（约 10 人天/需求），调整规则带来刷数
沟通prd等更多隐性成本。不利于业务规则和产品功能频繁更新，为了支撑连锁商家业务的快速扩张和精细化运营，提升商品管理效率和用户体验，需要建设。

因此：我建设了一个新的检索系统，并通过检索条件标签化的形式，化繁为简，检索侧进需要关注查询需要哪些标签即可。
